Common Mistakes in GATE DA AI Section

GATE Relevance: Critical
💡 Why This Page Matters

Knowing the correct concepts is only half the battle. Many students lose marks not because they don’t know the material, but because they fall into predictable traps. This page catalogues the most frequent mistakes by topic so you can recognise and avoid them on exam day.

Part A — Search Algorithm Mistakes

⚠ Mistake 1: Confusing Completeness vs Optimality

The mistake: Students treat “complete” and “optimal” as interchangeable, believing that a complete algorithm always returns the best solution.

Why it happens: Both words sound positive, so students conflate “will find a solution” with “will find the best solution.”

Correct understanding:

  • Complete — Guaranteed to find a solution if one exists (may not be the cheapest).
  • Optimal — Guaranteed to find the least-cost solution.

An algorithm can be complete but not optimal (e.g., DFS on a finite graph finds a solution but not necessarily the shortest one). Conversely, an incomplete algorithm may never terminate, so optimality is irrelevant.

Example — Mistake 1
Graph:  S → A (cost 1) → G (cost 10)
        S → B (cost 5) → G (cost 1)

❌ Wrong reasoning: “DFS is complete on this finite graph, so it will find the optimal path.”

DFS might explore S → A → G (total cost 11) first and return it.

✅ Correct: The optimal path is S → B → G (cost 6). DFS found a solution (complete ✔) but not the cheapest (not optimal ✘). Only UCS or A* (with admissible h) guarantees optimality here.

⚠ Mistake 2: Forgetting DFS Is Not Complete in Infinite Spaces

The mistake: Claiming DFS is always complete. Students forget the critical caveat: DFS can loop forever in infinite state spaces (or graphs with cycles and no visited check).

Why it happens: Textbook examples usually use small finite trees where DFS works fine, building a false sense of generality.

Correct understanding:

  • DFS is not complete in infinite-depth spaces — it may descend an infinite branch and never backtrack.
  • DFS is complete only in finite state spaces (with cycle detection or a depth limit).
  • Iterative Deepening DFS (IDDFS) fixes this: it is complete even in infinite spaces.
Example — Mistake 2
State space: S → A → B → C → D → … (infinite chain)
                          ¦
                          → G  (goal reachable from S via a different branch)

❌ Wrong: “DFS will find G since the graph contains a path to it.”

✅ Correct: If DFS explores the infinite branch S → A → B → C → … first, it will never backtrack to discover the branch leading to G. DFS is incomplete in this space. Use BFS or IDDFS instead.

⚠ Mistake 3: Wrong Complexity Calculations (Confusing b, d, m)

The mistake: Mixing up the three key parameters when computing time/space complexity of search algorithms.

Why it happens: The symbols look similar, and different textbooks sometimes use slightly different notation.

Correct definitions:

  • b = branching factor (max number of successors of any node)
  • d = depth of the shallowest goal node
  • m = maximum depth of the search tree (may be ∞)

Key complexities to memorise:

AlgorithmTimeSpaceUses which depth?
BFSO(bd)O(bd)d (shallowest goal)
DFSO(bm)O(bm)m (max depth)
IDDFSO(bd)O(bd)d
UCSO(b1+⌊C*/ε⌋)Samecost-based
Example — Mistake 3

GATE-style question: A search tree has branching factor b = 3, shallowest goal at depth d = 4, and maximum depth m = 10. What is the worst-case time complexity of DFS?

❌ Wrong answer: O(34) = O(81) — using d instead of m.

✅ Correct answer: O(310) = O(59049) — DFS may explore the entire tree to depth m before finding the goal.

⚠ Mistake 4: Assuming UCS = BFS (Ignoring Non-Uniform Costs)

The mistake: Treating Uniform-Cost Search and Breadth-First Search as identical algorithms.

Why it happens: When all edge costs are equal (say, 1), UCS behaves exactly like BFS. Students over-generalise from this special case.

Correct understanding:

  • BFS expands by depth level (number of edges from start). It is optimal only when all step costs are identical.
  • UCS expands by path cost g(n). It is optimal for any non-negative step costs (≥ ε > 0).
  • When costs vary, BFS can return a suboptimal solution.
Example — Mistake 4
Graph:
  S → A  cost 1
  S → B  cost 5
  A → G  cost 10
  B → G  cost 1

BFS order: S → expand level 1: {A, B} → expand level 2: A→G found! Path cost = 1 + 10 = 11.

UCS order: S(0) → A(1) → B(5) → B→G(6). Path cost = 5 + 1 = 6. ✅ Optimal!

❌ BFS returned cost 11. UCS returned cost 6. They are not the same when costs differ.

⚠ Mistake 5: Thinking A* with Inadmissible Heuristic Always Gives Wrong Answer

The mistake: Believing that if h(n) is inadmissible (overestimates), A* will always return a suboptimal solution.

Why it happens: The textbook states “A* is optimal if h is admissible.” Students read the contrapositive incorrectly: admissibility is a sufficient condition, not a necessary one.

Correct understanding:

  • An inadmissible heuristic means A* may return a suboptimal solution — but it does not guarantee a wrong answer.
  • If the overestimation does not affect the expansion order on the path to the goal, A* might still find the optimal solution by luck.
  • The key point: admissibility gives a guarantee. Without it, we lose the guarantee but don’t necessarily lose correctness in every instance.
Example — Mistake 5
Graph: S → A(1) → G(2)    and    S → G(5)
Inadmissible h: h(S)=10, h(A)=5, h(G)=0
True h*: h*(S)=3, h*(A)=2, h*(G)=0
h(A)=5 > h*(A)=2, so h is inadmissible.

A* trace:

  • S: f = 0 + 10 = 10. Expand S.
  • A: f = 1 + 5 = 6. G(via S): f = 5 + 0 = 5. Expand G(via S), cost = 5.

Wait — the optimal path S→A→G costs 3, but A* found cost 5. Here, inadmissibility did cause a mistake.

But consider: if h(A) = 3 (still > h*(A) = 2, so inadmissible), then f(A) = 1+3 = 4 < f(G via S) = 5, so A* would expand A first and find optimal cost 3. Same inadmissible heuristic structure, different outcome.

✅ Takeaway: Inadmissible h can cause suboptimal results, but doesn’t always do so. GATE may test this nuance.

Part B — Logic Mistakes

⚠ Mistake 6: Wrong CNF Conversion (Forgetting to Distribute ∨ over ∧)

The mistake: Students eliminate implications and push negations inward correctly, but then skip or misapply the distribution step needed to reach Conjunctive Normal Form.

Why it happens: The distribution law (A ∨ (B ∧ C) ≡ (A ∨ B) ∧ (A ∨ C)) can produce an exponential blowup, so students rush it or forget to apply it at all.

Correct CNF conversion steps:

  1. Eliminate → and ↔ using equivalences
  2. Push ¬ inward using De Morgan’s laws (and double negation)
  3. Distribute ∨ over ∧ until you have a conjunction of clauses (every clause is a disjunction of literals)
Example — Mistake 6

Convert to CNF: A ∨ (B ∧ C)

❌ Wrong: “It’s already in CNF.” (Student sees ∨ and ∧ and assumes it’s done.)

But A ∨ (B ∧ C) is not a conjunction of disjunctive clauses — it has ∨ at the top level with ∧ nested inside.

✅ Correct: Apply distribution:

A ∨ (B ∧ C)
≡ (A ∨ B) ∧ (A ∨ C)    ← distribute ∨ over ∧

Now it is CNF: two clauses (A ∨ B) and (A ∨ C).

Harder example: (P ∨ Q) ∨ (R ∧ S ∧ T)

≡ ((P ∨ Q) ∨ R) ∧ ((P ∨ Q) ∨ S) ∧ ((P ∨ Q) ∨ T)
≡ (P ∨ Q ∨ R) ∧ (P ∨ Q ∨ S) ∧ (P ∨ Q ∨ T)
⚠ Mistake 7: Confusing ∀ and ∃ Scope in Nested Quantifiers

The mistake: Students swap the order of quantifiers or assume they commute freely. The order of ∀ and ∃ matters.

Why it happens: In English, the difference between “everyone loves someone” and “someone is loved by everyone” feels subtle. In logic, the meanings are drastically different.

Correct understanding:

  • ∀x ∃y Loves(x, y) = For every person x, there exists some person y that x loves. (Each person has at least one loved one — possibly different for each person.)
  • ∃y ∀x Loves(x, y) = There exists a person y such that every person x loves y. (One specific person is loved by everyone.)
  • These are not equivalent. ∃y ∀x ⇒ ∀x ∃y, but not vice versa.
Example — Mistake 7

GATE-style question: Translate “Every student has a favourite teacher” into FOL.

❌ Wrong: ∃t ∀s FavTeacher(s, t) — this says there is one teacher who is the favourite of all students.

✅ Correct: ∀s ∃t FavTeacher(s, t) — for each student, there exists (possibly different) a favourite teacher.

Key rule: The quantifier on the “dependent” variable goes after (inside) the quantifier it depends on. Here, the teacher depends on the student, so ∃t goes after ∀s.

⚠ Mistake 8: Unification with Occurs Check Failures

The mistake: Attempting to unify a variable x with a term that contains x (e.g., unifying x with f(x)), producing an infinite structure.

Why it happens: Students mechanically apply the substitution {x/f(x)} without checking whether x appears inside the term being substituted.

Correct understanding:

  • Before binding x → t, always check: does x occur in t?
  • If x ∈ vars(t), unification fails (occurs check). The binding x/f(x) would create an infinite term f(f(f(…))).
  • GATE often tests this with predicates like P(x, f(x)) and P(y, y).
Example — Mistake 8

Unify: P(x, f(x)) with P(a, y)

Step 1: x = a ⇒ θ = {x/a}. Apply: P(a, f(a)) and P(a, y).

Step 2: y = f(a) ⇒ θ = {x/a, y/f(a)}. ✅ Success.

Now unify: P(x, f(x)) with P(y, y)

Step 1: x = y ⇒ θ = {x/y}. Apply: P(y, f(y)) and P(y, y).

Step 2: y = f(y)? Occurs check: y appears in f(y). ❌ FAIL.

❌ Wrong answer: θ = {x/y, y/f(y)} — this is invalid because it creates an infinite term.

✅ Correct answer: Unification fails.

⚠ Mistake 9: Wrong Application of Modus Ponens

The mistake: Applying Modus Ponens in the wrong direction — from the consequent to the antecedent — which is the fallacy of affirming the consequent.

Why it happens: Students remember “if A then B, and we know something, therefore something else” but get confused about which part must be known.

Correct rule (Modus Ponens):

Given:  A → B   and   A
Derive: B

Invalid (Affirming the Consequent):

Given:  A → B   and   B
Derive: A     ← INVALID!

Also distinguish from Modus Tollens (which IS valid):

Given:  A → B   and   ¬B
Derive: ¬A   ✔
Example — Mistake 9

Knowledge base: “If it rains, the ground is wet.” (Rain → Wet)

❌ Wrong reasoning: “The ground is wet. Therefore, it rained.” (Wet ⇒ Rain — affirming the consequent. The ground could be wet from a sprinkler.)

✅ Correct Modus Ponens: “It is raining. Therefore, the ground is wet.” (Rain, Rain→Wet ⇒ Wet ✔)

✅ Correct Modus Tollens: “The ground is NOT wet. Therefore, it is NOT raining.” (¬Wet, Rain→Wet ⇒ ¬Rain ✔)

Part C — Probability & Bayesian Network Mistakes

⚠ Mistake 10: Confusing Prior and Posterior

The mistake: Using P(Disease) when the question asks for P(Disease | Test+), or vice versa.

Why it happens: Students forget that evidence updates the probability. The word “probability” is used loosely in English, blurring the distinction between before and after observing evidence.

Correct understanding:

  • Prior P(H) = probability of hypothesis before seeing evidence.
  • Posterior P(H|E) = probability of hypothesis after seeing evidence E.
  • Bayes’ Theorem: P(H|E) = P(E|H) · P(H) / P(E)
  • The posterior is almost never equal to the prior (unless E is independent of H).
Example — Mistake 10

Disease prevalence: P(D) = 0.01. Test sensitivity: P(+|D) = 0.95. False positive rate: P(+|¬D) = 0.05.

Question: A person tests positive. What is the probability they have the disease?

❌ Wrong: “The test is 95% accurate, so P(D|+) ≈ 0.95.” (Confused the likelihood P(+|D) with the posterior P(D|+).)

✅ Correct: Apply Bayes’ Theorem:

P(D|+) = P(+|D) · P(D) / P(+)
P(+)   = P(+|D)P(D) + P(+|¬D)P(¬D)
       = 0.95 × 0.01 + 0.05 × 0.99
       = 0.0095 + 0.0495 = 0.059
P(D|+) = 0.0095 / 0.059 ≈ 0.161 (about 16%, NOT 95%!)
⚠ Mistake 11: Wrong Variable Elimination Order

The mistake: Eliminating variables in an arbitrary order without considering which order minimises intermediate factor sizes, or worse, eliminating a query variable or evidence variable.

Why it happens: Students learn the algorithm mechanically and forget the constraints on which variables must be preserved. Also, good elimination orderings require thought.

Correct approach:

  • Never eliminate the query variable(s) or evidence variables.
  • Eliminate only the hidden (non-query, non-evidence) variables.
  • Prefer eliminating variables that appear in the fewest factors first (min-degree heuristic) or that create the smallest new factors (min-fill heuristic).
  • A bad ordering is still correct — just much slower. But eliminating the wrong variable entirely changes the result.
Example — Mistake 11

Bayesian Network: A → B → C. Query: P(C | A = true). Hidden variable: B.

❌ Wrong: Student eliminates C (the query variable!) and computes P(B | A=true) instead.

✅ Correct:

P(C | A=true) = ∑B P(B | A=true) · P(C | B)
             = P(B=t|A=t)·P(C|B=t) + P(B=f|A=t)·P(C|B=f)

We sum out B (the hidden variable), keeping A fixed (evidence) and C as the result (query).

⚠ Mistake 12: Forgetting Conditional Independence ≠ Marginal Independence

The mistake: Assuming that if X and Y are conditionally independent given Z, then X and Y are also marginally independent (or vice versa).

Why it happens: The word “independent” is overloaded. Students don’t fully internalise that the conditioning context changes the independence relationship.

Correct understanding:

  • Conditional independence: P(X, Y | Z) = P(X | Z) · P(Y | Z). Written X ⊥ Y | Z.
  • Marginal independence: P(X, Y) = P(X) · P(Y). Written X ⊥ Y.
  • Neither implies the other in general!
  • Classic Bayesian Network example: In the “explaining away” (v-structure) pattern A → C ← B, A and B are marginally independent but not conditionally independent given C.
Example — Mistake 12

Bayesian Network (v-structure): Earthquake → Alarm ← Burglary

Earthquake and Burglary are independent causes of the Alarm.

  • Marginally: P(E, B) = P(E) · P(B) ✔ (independent before observing Alarm).
  • Conditionally given Alarm=true: If we know the alarm went off and then learn there was no earthquake, the probability of burglary increases (explaining away). So E and B are not conditionally independent given A.

❌ Wrong: “E ⊥ B, therefore E ⊥ B | A.”

✅ Correct: E ⊥ B is true, but E ⊥ B | A is false. Observing the common effect (Alarm) introduces a dependency between the causes.

⚠ Mistake 13: Not Normalising After Variable Elimination

The mistake: After summing out hidden variables, students report the raw factor values without normalising, yielding numbers that don’t sum to 1.

Why it happens: Variable elimination produces an unnormalised distribution. The normalisation step is easy to forget, especially under exam pressure.

Correct approach:

  • After eliminating all hidden variables, you get a factor φ(query).
  • The final step is to normalise: P(Q = q) = φ(q) / ∑q′ φ(q′).
  • If your answers don’t sum to 1, you almost certainly forgot to normalise.
Example — Mistake 13

After variable elimination, you obtain:

φ(C = true)  = 0.054
φ(C = false) = 0.126

❌ Wrong answer: P(C = true) = 0.054, P(C = false) = 0.126. (These sum to 0.18, not 1!)

✅ Correct: Normalise:

α = 1 / (0.054 + 0.126) = 1 / 0.18 ≈ 5.556
P(C = true)  = 0.054 / 0.18 = 0.30
P(C = false) = 0.126 / 0.18 = 0.70
                            Sum = 1.00 ✔

Part D — General Exam Mistakes

⚠ Mistake 14: Time Management — Spending Too Long on One Question

The mistake: Getting stuck on a single hard question (often a 2-mark question) and spending 10–15 minutes on it, leaving insufficient time for easier questions worth the same or more marks.

Why it happens: Test anxiety, sunk-cost fallacy (“I’ve already spent 5 minutes, I must finish it”), or underestimating the remaining questions.

Correct approach:

  • GATE DA has 65 questions in 180 minutes — roughly 2.8 minutes per question on average.
  • 1-mark questions: aim for ≤ 2 minutes. 2-mark questions: aim for ≤ 4 minutes.
  • If you’re stuck after 3–4 minutes, mark for review and move on. Return to it later if time permits.
  • Answering three easy 1-mark questions (3 marks) is better than struggling with one hard 2-mark question.

First pass0–90 minAttempt all questions, skip hard ones quickly
Second pass90–150 minReturn to marked/skipped questions
Review150–180 minVerify answers, check NAT rounding, fix bubbles
⚠ Mistake 15: MCQ vs MSQ Confusion (Partial Marking)

The mistake: Not understanding the different marking schemes for MCQ (Multiple Choice Questions) and MSQ (Multiple Select Questions), leading to wrong risk-reward decisions.

Why it happens: Students don’t read the instructions carefully, or they apply MCQ strategy to MSQ and vice versa.

Correct understanding:

TypeCorrectWrong/PartialUnansweredNegative Marking?
MCQ (1 correct)Full marks−1/3 (1-mark) or −2/3 (2-mark)0Yes
MSQ (1+ correct)Full marks0 (partial or wrong)0No
NAT (numerical)Full marks00No

Key strategy implications:

  • For MCQ: guessing randomly has negative expected value. Eliminate at least one option before guessing.
  • For MSQ: no penalty for wrong answers, but partial correct answers get 0 marks. You must select all correct options and no wrong ones. Always attempt MSQ since there’s no negative marking.
  • For NAT: always enter an answer — no penalty for wrong attempts.
Example — Mistake 15

Scenario (MCQ, 2 marks): You’re unsure between options A and C. You’ve eliminated B and D.

❌ Wrong strategy: Skip it (“I’m not sure”). Expected value of skipping = 0.

✅ Better strategy: With 2 remaining options, you have a 50% chance of getting +2 and 50% chance of getting −2/3. Expected value = 0.5 × 2 + 0.5 × (−2/3) = 1 − 0.333 = +0.667. Always guess when you can eliminate options!

Scenario (MSQ, 2 marks): You’re sure about options A and B, unsure about C.

❌ Wrong: Select A, B, and C “just to be safe” (if C is wrong, you get 0 marks despite A and B being right).

✅ Correct analysis: If you’re > 50% confident C is correct, include it (partial answers get 0 anyway). If unsure, leave C out only if A+B is the complete correct set. In MSQ, you need exactly the right set.

⚠ Mistake 16: NAT Rounding Errors

The mistake: Entering a NAT (Numerical Answer Type) response with incorrect rounding, or entering it in the wrong format, causing the system to mark a correct solution as wrong.

Why it happens: Students round intermediate results too aggressively, or they’re unsure whether to round to 2, 3, or 4 decimal places.

Correct approach:

  • GATE typically accepts NAT answers within a range (e.g., ±0.01 or ±1% of the correct answer).
  • Never round intermediate calculations. Keep at least 4–5 decimal places throughout, and round only the final answer.
  • If the question says “round to 2 decimal places,” do exactly that. Otherwise, enter the most precise value you calculated (2–4 decimal places).
  • Check the format: some NAT questions expect integers, others expect decimals. Read the instruction under the answer box.
Example — Mistake 16

Question: Calculate the entropy H(X) where P(X=0) = 0.7, P(X=1) = 0.3. Answer up to 2 decimal places.

❌ Wrong approach (premature rounding):

log2(0.7) ≈ −0.51 (rounded too early)
log2(0.3) ≈ −1.74 (rounded too early)
H = −(0.7 × −0.51 + 0.3 × −1.74)
  = −(−0.357 − 0.522) = 0.88

✅ Correct approach (full precision intermediate steps):

log2(0.7) = −0.51457 (keep full precision)
log2(0.3) = −1.73697
H = −(0.7 × (−0.51457) + 0.3 × (−1.73697))
  = −(−0.36020 − 0.52109)
  = −(−0.88129) = 0.88129
  ⇒ Rounded: 0.88

In this case the answers happen to match, but with more complex calculations premature rounding can push you outside the accepted range. Always keep full intermediate precision.

Summary — Quick Reference Checklist

📋 Before-Exam Checklist

Print or memorise this list. Skim it the night before the exam.

Search Algorithms

  • ☑ Complete ≠ Optimal. Know the difference.
  • ☑ DFS is not complete in infinite spaces.
  • ☑ b = branching factor, d = shallowest goal, m = max depth. DFS uses m, BFS uses d.
  • ☑ UCS ≠ BFS when edge costs differ.
  • ☑ Inadmissible h may still give optimal results — but removes the guarantee.

Logic

  • ☑ CNF requires distributing ∨ over ∧. Don’t stop early.
  • ☑ ∀x ∃y ≠ ∃y ∀x. Quantifier order matters.
  • ☑ Occurs check: cannot unify x with f(x).
  • ☑ Modus Ponens: A, A→B ⇒ B. Not: B, A→B ⇒ A.

Probability & Bayesian Networks

  • ☑ P(H|E) ≠ P(E|H). Always apply Bayes’ Theorem.
  • ☑ Never eliminate query or evidence variables.
  • ☑ Conditional independence ≠ marginal independence (explaining away).
  • ☑ Always normalise VE results so probabilities sum to 1.

General Exam Strategy

  • ☑ ~2.8 min per question. Move on after 4 min max.
  • ☑ MCQ has negative marking; MSQ and NAT do not.
  • ☑ For NAT, keep full precision in intermediate steps. Round only the final answer.
