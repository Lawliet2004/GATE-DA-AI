Partial-Order Planning (POP)

GATE Relevance: Medium

🎯 Learning Objectives

  • Contrast total-order and partial-order plans
  • Explain the least commitment strategy
  • Identify the four components of a partial-order plan
  • Trace the POP algorithm step by step
  • Resolve threats using promotion and demotion
  • Understand causal link protection

1. Total-Order vs. Partial-Order Plans

📚 Total-Order Plan

A total-order plan is a fully ordered sequence of actions: a1 → a2 → … → an. Every pair of actions has a definite ordering. This is what forward or backward state-space search produces.

📚 Partial-Order Plan

A partial-order plan specifies only the ordering constraints that are necessary. Two actions are left unordered if there is no reason one must precede the other. This gives more flexibility in execution.

FeatureTotal-OrderPartial-Order
OrderingAll actions linearly orderedOnly necessary orderings specified
FlexibilitySingle execution sequenceMultiple valid linearisations
Search spaceSequences of actionsSets of constraints (plan space)
StrategyEager commitmentLeast commitment
Handles Sussman Anomaly?Only with backtrackingYes, naturally

2. The Least Commitment Strategy

📚 Least Commitment Principle

Delay making ordering or binding decisions until they are forced by a constraint. This avoids premature commitment that could lead to backtracking. POP embodies this principle by leaving actions unordered whenever possible.

💡 Benefit

By keeping the plan partially ordered, the planner avoids committing to an ordering that might later prove wrong. This is precisely why POP can handle the Sussman Anomaly without difficulty — it doesn’t force sub-goal actions into a fixed sequence.

3. Components of a Partial-Order Plan

A partial-order plan π is a tuple ⟨A, O, L, OC⟩:

📚 Plan Components

  • A — Actions: the set of actions in the plan, always including two special actions:
    • Start (a0): no preconditions; its effects are the initial state fluents
    • Finish (a∞): its preconditions are the goal fluents; no effects
  • O — Ordering Constraints: a set of pairs ai ≺ aj meaning “ai must execute before aj”. Always includes Start ≺ Finish.
  • L — Causal Links: a set of triples ai ⟶p aj meaning “action ai achieves precondition p for action aj”.
  • OC — Open Preconditions: preconditions of actions in A that are not yet supported by any causal link.

🔬 Initial Plan (Before Any Refinement)

A = {Start, Finish}
O = {Start ≺ Finish}
L = {} (empty)
OC = all preconditions of Finish = the goal literals

4. The POP Algorithm

Partial-Order Planning (POP) Algorithm

function POP(initial_state, goal):
    plan ← initial plan with Start, Finish
    // Start effects = initial state; Finish preconditions = goal
    open_preconditions ← preconditions of Finish

    while open_preconditions ≠ ∅:
        select an open precondition p of some action a_need

        // Step 1: Find an action a_add that achieves p
        choose a_add such that p ∈ Add(a_add)
        // a_add can be an existing action in the plan OR a new action

        if a_add is new:
            add a_add to plan.actions
            add Start ≺ a_add ≺ Finish to plan.orderings
            add preconditions of a_add to open_preconditions

        // Step 2: Add causal link and ordering
        add causal link a_add →p→ a_need
        add ordering a_add ≺ a_need

        // Step 3: Resolve threats
        for each action a_threat in plan.actions:
            if p ∈ Del(a_threat) and
               a_threat could be ordered between a_add and a_need:
                resolve threat by:
                  PROMOTION:  add ordering a_need ≺ a_threat
                  or DEMOTION: add ordering a_threat ≺ a_add
                // Choose whichever does not create a cycle

        remove p from open_preconditions

    if plan is consistent (no cycles, no threats):
        return plan
    else:
        backtrack and try alternative choices

💡 Non-deterministic Choices

The POP algorithm has two key choice points requiring backtracking: (1) which open precondition to work on, and (2) which action to use to achieve it. Good heuristics at these choice points dramatically affect performance.

5. Threats and Their Resolution

📚 Threat (Clobberer)

An action at threatens a causal link ai ⟶p aj if:

  1. p ∈ Del(at) — the action deletes the protected proposition
  2. at could possibly be ordered between ai and aj (i.e., the orderings ai ≺ at ≺ aj are consistent with existing constraints)

Two standard resolution strategies:

📚 Promotion

Add the ordering constraint aj ≺ at — force the threatening action to come after the consumer. This “promotes” aj ahead of the threat.

📚 Demotion

Add the ordering constraint at ≺ ai — force the threatening action to come before the producer. This “demotes” at behind the link’s origin.

⚠ When Neither Works

If both promotion and demotion would create an ordering cycle, nor can an alternative achiever be found, the algorithm must backtrack to an earlier choice point. POP is sound and complete with proper backtracking.

🔬 Threat Example

Suppose we have causal link: MoveFromTable(B,C) ⟶Clear(B) MoveFromTable(A,B)

A threatening action might be: MoveFromTable(D,B) which deletes Clear(B).

  • Promotion: order MoveFromTable(A,B) ≺ MoveFromTable(D,B) — use B before D covers it
  • Demotion: order MoveFromTable(D,B) ≺ MoveFromTable(B,C) — cover B with D first, but then B’s move provides Clear(B) later

6. Causal Link Protection

Causal links serve as protected conditions. The invariant is:

📜 Causal Link Invariant

For every causal link ai ⟶p aj in the plan, no action ak with p ∈ Del(ak) can be ordered such that ai ≺ ak ≺ aj.

This invariant guarantees that when any valid linearisation of the partial order is executed, the precondition p will still hold when aj is reached. Maintaining this invariant is the primary role of threat resolution.

7. Worked Example: Sussman Anomaly via POP

🔬 POP Solving the Sussman Anomaly

Initial: On(C,A), OnTable(A), OnTable(B), Clear(C), Clear(B)

Goal: On(A,B) ∧ On(B,C)

Step 0: Initial plan

  • Actions: {Start, Finish}
  • Start effects: On(C,A), OnTable(A), OnTable(B), Clear(C), Clear(B)
  • Finish preconditions: On(A,B), On(B,C)
  • Open: {On(A,B)@Finish, On(B,C)@Finish}

Step 1: Achieve On(A,B)

  • Add new action: MoveFromTable(A,B)
  • Causal link: MoveFromTable(A,B) ⟶On(A,B) Finish
  • New open preconditions: OnTable(A), Clear(A), Clear(B) for MoveFromTable(A,B)

Step 2: Achieve On(B,C)

  • Add new action: MoveFromTable(B,C)
  • Causal link: MoveFromTable(B,C) ⟶On(B,C) Finish
  • New open: OnTable(B), Clear(B), Clear(C) for MoveFromTable(B,C)
  • ⚠ Threat detected: MoveFromTable(A,B) deletes Clear(B), threatening link Start ⟶Clear(B) MoveFromTable(B,C)
  • Resolution (demotion): order MoveFromTable(B,C) ≺ MoveFromTable(A,B)

Step 3: Achieve Clear(A) for MoveFromTable(A,B)

  • Clear(A) is not in Start’s effects (C is on A!). Need an action.
  • Add: MoveToTable(C,A) with Add={OnTable(C), Clear(A)}, Del={On(C,A)}
  • Causal link: MoveToTable(C,A) ⟶Clear(A) MoveFromTable(A,B)
  • Open: On(C,A)@MoveToTable(C,A), Clear(C)@MoveToTable(C,A) — both satisfied by Start

Step 4: Link remaining open preconditions to Start

  • OnTable(A) ← Start ✓, OnTable(B) ← Start ✓, Clear(B) ← Start ✓, Clear(C) ← Start ✓

Final Partial Order:

  • Start ≺ MoveToTable(C,A) ≺ MoveFromTable(B,C) ≺ MoveFromTable(A,B) ≺ Finish

One valid linearisation: MoveToTable(C,A) → MoveFromTable(B,C) → MoveFromTable(A,B) ✓

Summary

  • Partial-order plans only impose necessary ordering constraints — embodying the least commitment principle.
  • A plan is represented as ⟨Actions, Orderings, Causal Links, Open Preconditions⟩.
  • The POP algorithm refines the plan by selecting open preconditions, finding achievers, and resolving threats.
  • Threats occur when an action could delete a causally-linked proposition; they are resolved by promotion (consumer before threat) or demotion (threat before producer).
  • POP naturally handles problems like the Sussman Anomaly that defeat linear planners.
