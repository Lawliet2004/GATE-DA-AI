📝 Planning — Quick Notes

GATE Cheatsheet

1. STRIPS Notation Summary

ComponentDescription
StateSet of ground atomic propositions (fluents) that are true. CWA: everything else is false.
PreconditionsConjunction of positive literals that must be true for the action to be applicable.
Add List (effects+)Literals that become true after the action. Also called positive effects.
Delete List (effects−)Literals that become false after the action. Also called negative effects.
Result(s, a)(s − Del(a)) ∪ Add(a) — delete first, then add.
ApplicableAction a is applicable in state s iff Precond(a) ⊆ s.
Frame AssumptionFluents not in Add or Delete lists are unchanged.

2. Planning Algorithm Comparison

AlgorithmApproachDirectionComplete?Optimal?Key Feature
Forward Search State-spaces0 → goalYesWith A* Simple; large branching factor
Backward Search State-spacegoal → s0YesWith A* Only relevant actions; smaller branching

POP Plan-spaceRefinementYesNo* Least commitment; handles sub-goal interaction

* POP can be made optimal with appropriate cost tracking, but standard POP is satisficing.

3. Mutex Types (GraphPlan)

Mutex TypeLevelConditionStatic?
Inconsistent EffectsActionAdd(a1) ∩ Del(a2) ≠ ∅Yes
InterferenceActionDel(a1) ∩ Precond(a2) ≠ ∅Yes
Competing NeedsActionPreconditions of a1, a2 are mutex in PiNo
Inconsistent SupportPropositionAll achiever pairs are mutex in AiNo

⚠ Remember

Static mutexes (inconsistent effects, interference) never disappear. Derived mutexes (competing needs, inconsistent support) may vanish as the graph grows — mutexes shrink monotonically.

5. POP Quick Reference

Plan = ⟨A, O, L, OC⟩

  • A — Actions (includes Start & Finish)
  • O — Ordering constraints (ai ≺ aj)
  • L — Causal links (ai ⟶p aj)
  • OC — Open preconditions (not yet supported)

Threats: action at threatens link ai ⟶p aj if p ∈ Del(at) and at can be between ai and aj.

Resolution:

  • Promotion: aj ≺ at (consumer before threat)
  • Demotion: at ≺ ai (threat before producer)

6. Key Definitions

TermDefinition
Closed-World AssumptionAny fluent not stated is false.
Frame ProblemHow to represent that most things don’t change when an action occurs.
STRIPS AssumptionOnly add/delete list fluents change; everything else persists.
Sussman AnomalyA problem where linear planners fail because sub-goals interact negatively.
Least CommitmentDelay ordering/binding decisions until forced. POP embodies this principle.
Causal Linkai ⟶p aj: action ai achieves precondition p for aj.
Level-OffPlanning graph property: layers stabilise (Pk = Pk+1, same mutexes). Guarantees GraphPlan termination.
Relaxed PlanPlan in the problem with delete effects removed. Used for heuristic computation.

8. Common GATE Questions on Planning

  • Given a STRIPS operator, compute the resulting state after applying it to a given state. Just apply: Result(s,a) = (s − Del(a)) ∪ Add(a).
  • Identify mutex actions/propositions in a given planning graph layer. Check all four mutex conditions systematically.
  • Sussman Anomaly: explain why a given ordering fails; provide the correct interleaved plan.
  • Determine if an action is applicable in a given state. Check Precond(a) ⊆ s.
  • POP threat resolution: given a causal link and a threatening action, specify promotion and demotion orderings.
  • Planning graph layer construction: build the next proposition/action layer and identify new mutexes.
  • True/False conceptual questions: “STRIPS goal-stack planning is complete” (False), “GraphPlan always terminates” (True), etc.

9. Tips & Common Mistakes

  • When applying STRIPS actions, always delete before adding. If a fluent is in both lists, it ends up true.
  • In POP, Start has no preconditions and its effects = initial state. Finish has the goal as preconditions and no effects.
  • Don’t confuse interference (delete vs. precondition) with inconsistent effects (delete vs. add).
  • The planning graph over-approximates reachable propositions. Just because a proposition appears doesn’t mean it’s actually achievable — mutex checking tightens this.
  • In the blocks world, always check Clear predicates carefully. Forgetting to update them is the most common error.
  • GraphPlan finds the shortest parallel plan (minimum number of time steps), not necessarily the fewest total actions.
